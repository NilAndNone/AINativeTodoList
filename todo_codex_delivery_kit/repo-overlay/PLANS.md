# ExecPlans / 可执行计划规范

当任务涉及以下任一情况时，必须先写或先对齐 ExecPlan，再进入实现：

- 跨多个模块的重构
- MCP 工具面变化
- 数据契约变化
- 代码仓 / 数据仓边界变化
- 测试框架或目录结构调整
- 任意可能影响已有 workflow 的改动

## ExecPlan 要求

1. 必须自包含。只看计划本身和当前工作树，一个新加入的人也能继续推进。
2. 必须写清：
   - 背景
   - 范围
   - 非范围
   - 输入输出
   - 变更文件
   - 验收命令
   - 风险与回滚
3. 计划是活文档。实现过程中发现事实变化，先更新计划，再继续实现。
4. 遇到长任务时，用里程碑推进，不要把整个世界一次改完。

## 对本仓库的补充要求

- 任何写操作都必须保留 `plan -> confirm -> apply` 语义
- `today.md` 仍是核心驾驶舱
- `id` 是稳定主键
- 配置优先进入数据仓，而不是写死进代码
- 先兼容旧 CLI，再替换旧逻辑
