先把当前系统拍成一张清晰的“术前 X 光片”。

### 要做什么

1. 记录当前仓库结构
2. 记录当前核心命令
3. 跑现有测试
4. 记录已知限制：
   - 项目映射写死
   - 已有任务只允许改状态/备注
   - 数据与代码耦合
   - Codex 没有正式 plugin/service 方案

### 建议改动

- 新增 `todo_codex_delivery_kit/` 目录
- 不改生产逻辑
- 如需新文档，只新增 `docs/baseline.md` 一类文件

### 验收命令

```bash
python3 -m pytest tests/
```

### 验收标准

- 测试全绿
- 没有破坏现有 CLI
- 形成 baseline 记录

## 阶段 1：分仓与配置

### 目标

把“数据仓在哪里”“项目映射是什么”“close-day 是否自动 push”这些东西，从代码里剥出来。

### 应新增内容

代码仓建议新增：

- `src/ainative_todo_service/config.py`
- `src/ainative_todo_service/doctor.py`
- `scripts/split_repo.py`（可从示例脚本演进）
- `docs/runtime-config.md`

数据仓建议新增：

- `todo.config.toml`

本机建议新增：

- `~/.config/ainative-todo/config.toml`

### 必须落地的行为

- code repo 能从本机 config 找到 data repo
- data repo 内有版本化配置
- 旧 CLI 能继续通过 `--root <data_repo>` 工作
- 示例 split 脚本能把旧仓数据拆出去

### 建议验收命令

```bash
python3 -m ainative_todo_service.doctor --config ~/.config/ainative-todo/config.toml
python3 scripts/todo_workflow.py --root /ABS/PATH/TO/DATA-REPO generate-today --date 2026-03-04
```

### 验收标准

- `doctor` 输出 code repo、data repo、data config 路径
- 指向数据仓的 `generate-today` 可以运行
- 代码仓不需要内置你的私人绝对路径

## 阶段 2：preview/apply 基础设施

### 目标

建立统一的“先预演，再落盘”的基础设施。

### 应新增内容

- `preview_runner.py`
- `operations.py`
- `contracts.py`

### 关键实现建议

- 通过临时目录复制 data repo
- 在临时目录执行写操作
- 计算 temp tree 与 real tree 的 unified diff
- preview 阶段**绝不修改真实数据仓**
- apply 阶段复用相同操作真正落盘

### 第一批接管的动作

优先把已有 CLI 能干的都接进 preview/apply：

- `generate_today`
- `close_day`
- `generate_weekly`
- `generate_monthly`
- `generate_quarterly`
- `generate_halfyear`
- `generate_projects`

### 建议验收命令

```bash
python3 -m pytest tests/
python3 -m pytest tests/test_preview_runner.py
```

### 人工验收点

- preview 后，真实数据仓没有变化
- apply 后，变化与 preview 看到的 diff 一致
- diff 中能清楚看到 `today.md`、`daily/`、`projects/`、`reports/` 的变化

## 阶段 4：MCP 写入动作层

### 目标

把“自然语言维护 todo”的核心能力真正接进来。

### 强制约束

- 一律 `plan -> confirm -> apply`
- 所有写操作都返回 diff
- 如果缺字段，返回 `missing_fields`
- skill/agent 只追问缺的，不重复问已经知道的

### 应支持的动作

必须：

- `start_day`
- `update_task`
- `add_task`
- `mark_done`
- `mark_blocked`
- `unblock_task`
- `change_priority`
- `change_due_date`
- `cancel_task`
- `close_day`
- `generate_report`
- `rebuild_projects`

可选：

- `change_project`

### 当前最大契约升级

你当前 `close-day` 逻辑只允许已有任务改 `状态` 和 `备注`。这一阶段必须推动契约升级：

- 已有任务至少允许改：
  - `title`
  - `project`
  - `priority`
  - `due_date`
  - `deliverable`
  - `status`
  - `notes`
- `id` 仍是稳定主键
- 所有改动要有校验与回显 diff

### 建议验收用例

#### 用例 1：更新任务进展

- 输入：“把 UTC 那个性能优化任务改成 blocked，备注改成等依赖方接口”
- 输出：
  - 若有多个候选，先返回 choices
  - 若唯一候选，直接给出 diff preview
- 确认后 apply

#### 用例 2：新增任务

- 输入：“新增一个 MISC 任务，明天前整理周会纪要”
- 如果缺 `priority` / `deliverable` / `status`，先补问
- 信息齐全后给 preview diff

#### 用例 3：修改既有任务标题和 DDL

- 输入：“把 AGE 那个阶段性计划标题改成阶段性计划 v2，DDL 改到周五”
- 必须能走通

### 验收标准

- 自然语言能驱动主要写入动作
- 任何写入都不会绕过确认
- 候选歧义能被工具层表达出来，而不是让 skill 瞎猜

## 阶段 5：skill + workflow + docs

### 目标

让 Codex 真正知道怎么用这套系统，而不是每次都靠临场发挥。

### 需要落地

- repo skill：`.agents/skills/todo-local-workflow/`
- `SKILL.md`
- `references/`
- 人类版 workflow
- agent 版 workflow
- 可选 `agents/openai.yaml`

### skill 中必须写清的规则

- 哪些意图触发
- 哪些工具优先使用
- 任何写入先调用 plan
- 看到 `missing_fields` 就追问
- 看到 `ready_for_confirm` 就展示 diff
- 用户没确认就不能 apply
- `close_day` 后允许自动 git commit + push（仅数据仓）

### 验收标准

- Codex 能显式触发 skill
- skill 文本与工具合同一致
- 人类版 / agent 版没有互相打架

## 阶段 6：迁移、硬化、收尾

### 目标

把“能 demo”变成“能长期用”。

### 建议完成项

- 数据迁移脚本完善
- e2e fixture
- close-day 自动 git commit + push
- 错误提示统一
- readme / runbook / rollback 文档
- test matrix

### 必做端到端场景

1. 开工生成 `today.md`
2. 白天更新一个已有任务
3. 新增一个临时任务
4. 生成周报
5. 收工归档
6. 自动提交并推送数据仓

### 最终验收命令建议

```bash
python3 -m pytest tests/
python3 -m pytest tests/test_e2e_workflow.py
```

### 最终完成定义

- 代码仓 / 数据仓分离
- Codex 可调用本地 service
- 能用自然语言完成主要操作
- 所有写入先 diff、后确认、再 apply
- 收工自动 commit + push
