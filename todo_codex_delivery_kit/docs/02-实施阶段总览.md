# 02 实施阶段总览

## 总体原则

- 每个阶段都必须有**明确边界**
- 每个阶段都必须有**验收命令**
- 没通过验收，不进下一阶段
- 允许 Codex 自主补齐实现细节，但**不允许跳阶段偷跑**

## 阶段表

| 阶段 | 名称 | 核心目标 | 主要输出 | 验收关键词 |
| --- | --- | --- | --- | --- |
| 0 | 基线冻结 | 保住当前可回滚状态 | baseline 说明、测试全绿 | 当前脚本和测试不被破坏 |
| 1 | 分仓与配置 | 代码仓 / 数据仓解耦 | 数据仓配置、runtime config、doctor、split 脚本 | 旧 CLI 可通过 `--root` 正常驱动数据仓 |
| 2 | preview/apply 基础设施 | 所有写入可先看 diff | preview runner、统一 diff、legacy operation wrappers | preview 不落盘，apply 落盘且 diff 一致 |
| 3 | MCP 只读层 | 先让 Codex 有可观察上下文 | doctor / overview / read tools、server skeleton | Codex 可发现并调用只读工具 |
| 4 | MCP 写入动作层 | 真正把自然语言维护任务接进来 | plan/apply、缺失字段补问、字段编辑能力 | 更新/新增/完成/阻塞/DDL/优先级可走 preview/apply |
| 5 | skill + workflow + docs | 让 Codex / Claude 真正会用 | repo skill、SOP、人类版与 agent 版文档 | skill 能显式触发，流程一致 |
| 6 | 迁移、硬化、收尾 | 端到端可用、能交接 | e2e 测试、迁移脚本完善、close-day git 自动化 | 从开工到收工全链路通过 |

## 推荐里程碑

### 里程碑 A：先活下来

完成阶段 0、1、2 之后，你就已经拥有：

- 独立数据仓
- preview/apply
- 与现有脚本兼容的工具层

这已经能明显降低风险。

### 里程碑 B：Codex 真能干活

完成阶段 3、4、5 之后，你会得到：

- Codex 可调用的本地工具
- skill 触发
- 自然语言更新/新增/收工/报表
- 统一 SOP

### 里程碑 C：可以长期维护

完成阶段 6 之后，系统才算“能长期服役”，不然很容易又长成第二团毛线球。

## 各阶段的完成定义

### 阶段 0 完成定义

- 当前仓库测试全绿
- 有一份 baseline 说明
- Codex 只做观察和记录，不做架构改动

### 阶段 1 完成定义

- 代码仓不再依赖固定路径
- runtime config 可解析本机数据仓路径
- 数据仓配置有样例文件
- 示例 split 脚本可复制现有数据目录结构

### 阶段 2 完成定义

- `generate-today`
- `close-day`
- `generate-weekly/monthly/quarterly/halfyear`
- `generate-projects`

都能：

- preview
- 展示 diff
- apply

### 阶段 3 完成定义

至少有这些只读能力：

- `doctor`
- `get_overview`
- `get_today_markdown`
- `get_task_candidates`

### 阶段 4 完成定义

至少有这些写能力：

- `start_day`
- `update_task`
- `add_task`
- `mark_done`
- `mark_blocked`
- `unblock_task`
- `change_priority`
- `change_due_date`
- `cancel_task`
- `close_day`
- `generate_report`
- `rebuild_projects`

并且都遵守：

- 缺失字段补问
- preview before write
- explicit confirm before apply

### 阶段 5 完成定义

- repo skill 落地
- Codex 可以从 `.agents/skills` 发现 skill
- 人类版 / agent 版流程文档齐备

### 阶段 6 完成定义

- e2e 用例通过
- close-day 自动 `commit + push`
- 文档写清部署和回滚办法
