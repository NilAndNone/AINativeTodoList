# 01 目标架构

## 1. 现状诊断

基于你当前仓库，关键现状是：

- `scripts/todo_workflow.py` 是单体 CLI，负责 today、归档、周期报告、项目页
- `data/tasks.csv` 是唯一事实源
- `today.md` 是当天驾驶舱
- 项目映射写死在代码里
- `close-day` 目前只允许改已有任务的 `状态` 和 `备注`
- skill 只偏向 Claude，Codex 没有同等级落地方案
- 数据和代码还在同一个仓里

这些点没有谁是“错”的，只是版本长大之后，开始出现经典的系统学喜剧：**最开始最省事的决定，后来都来收租**。

## 2. 目标状态

目标不是“造一个很炫的 AI 平台”，而是：

- 你对 agent 说自然语言
- agent 通过**统一本地 plugin/service**
- 去读写**独立数据仓**
- 所有写操作都先给你 **diff**
- 你确认后才 apply
- `today.md` 仍然是工作台
- Claude Code / Codex 共享同一执行内核

## 3. 推荐总架构

```text
┌────────────────────────────┐

│ - 自然语言交互              │

             │
             │ MCP / local tool calls
             ▼
```

## 4. 核心设计原则

### 4.1 写操作必须两段式

统一要求：

1. `plan / preview`
2. 用户确认
3. `apply`

任何会落盘的动作都不能偷偷越过这层。包括：

- 开工（生成 `today.md`）
- 更新任务
- 新增任务
- 修改优先级 / DDL / title / project / deliverable
- 取消 / 阻塞 / 解除阻塞
- 收工归档
- 生成报告
- 重建项目页

### 4.2 `today.md` 继续保留核心地位

你已经明确：agent 仍然围绕 `today.md` 工作。所以目标不是取消它，而是：

- 保留它的人类可读驾驶舱角色
- 把它纳入 tool/service 的受控写入模型
- 让它既能被生成，也能被 agent 维护，也能被收工同步

### 4.3 先兼容旧 CLI，再重构内核

v1 不追求一次性重写所有逻辑。推荐路线：

- 第一阶段先把旧 CLI 通过 `--root` 包起来
- 先得到分仓、preview/apply、基础工具化
- 再逐步把写操作从“编辑 today + close-day 同步”演进成更稳的 service 操作

### 4.4 配置放到数据仓

项目映射、路径、git 策略、默认值，都进入数据仓版本化配置：

- 让代码仓通用
- 让数据仓决定“你的具体世界长什么样”
- 减少硬编码和分叉修改

## 5. v1 最终边界

v1 要做到：

- 分仓
- 可定位数据仓
- preview/apply
- 兼容旧命令
- MCP 工具层
- Codex skill
- 人类版 / agent 版 workflow
- 收工自动 commit + push
- 周 / 月 / 季 / 半年报
- 项目页重建
- 可通过自然语言改已有任务主要字段

v1 暂时不要求：

- 把 CSV 立刻换成数据库
- 做远程服务部署
- 做多用户权限系统
- 做 Web UI
